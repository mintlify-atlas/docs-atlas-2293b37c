---
title: Running the Keeper Bot
description: Guide to running the COW Fee Module keeper bot for automated fee collection
---

The keeper bot automatically monitors token balances in the Settlement contract, creates swap orders on CoW Protocol, and executes the `drip()` function to collect fees.

## Prerequisites

- Node.js and Yarn installed
- Private key for the keeper address (set as `PRIVATE_KEY` environment variable)
- RPC URL for the target network
- Deployed COW Fee Module address

<Warning>
The keeper private key must match the `keeper` address configured in the module. The bot will verify this on startup and exit if there's a mismatch.
</Warning>

## CLI Options

The keeper bot accepts the following command-line options:

| Option | Description | Default | Required |
|--------|-------------|---------|----------|
| `--network <network>` | Network to run on: `mainnet`, `gnosis`, `arbitrum`, `base`, `sepolia`, `avalanche`, `polygon`, `bnb`, `linea`, `plasma`, `ink` | `mainnet` | No |
| `--rpc-url <rpc-url>` | Custom RPC URL (overrides network default) | Network default | No |
| `--module <module>` | COWFeeModule contract address | - | Yes |
| `--max-orders <max-orders>` | Maximum number of orders to place in single drip call | `250` | No |
| `--buy-amount-slippage-bps <bps>` | Slippage tolerance in basis points to add to quoted buyAmount | `100` (1%) | No |
| `--token-list-strategy <strategy>` | Strategy for getting token list: `explorer` or `chain` | `explorer` | No |
| `--lookback-range <n>` | Number of blocks to look back for Trade events (chain strategy only) | `1000` | No |
| `--max-fee-per-gas <gwei>` | Override maxFeePerGas transaction parameter (in GWEI) | Dynamic | No |
| `--max-priority-fee-per-gas <gwei>` | Override maxPriorityFeePerGas transaction parameter (in GWEI) | Dynamic | No |
| `-c, --confirm-drip` | Ask for confirmation before executing drip | `false` | No |
| `-h, --help` | Display help information | - | No |

## Token List Strategies

The keeper supports two strategies for discovering which tokens to swap:

### Explorer Strategy (Default)

Queries block explorer APIs to get the current token balances:

- **Mainnet**: Uses [Ethplorer API](https://ethplorer.io)
- **Gnosis**: Uses [Blockscout API](https://gnosis.blockscout.com)
- **Other networks**: Must use `chain` strategy

**Pros**: Fast, doesn't require many RPC calls

**Cons**: Limited to networks with supported explorers

### Chain Strategy

Scans on-chain `Trade` events from the Settlement contract to discover tokens:

- Looks back N blocks (default 1000, configurable with `--lookback-range`)
- Extracts all sellToken and buyToken addresses from Trade events
- Works on any network with RPC access

**Pros**: Works on all networks, no external API dependencies

**Cons**: Requires more RPC calls, slower for large lookback ranges

<Tip>
Use the `explorer` strategy when available for better performance. Use `chain` strategy for networks without explorer API support or when you need precise control over token discovery.
</Tip>

<Step title="Monitor execution">
The keeper will:

1. Load configuration from the module contract
2. Verify the keeper address matches the private key
3. Discover tokens using the selected strategy
4. Query CoW Protocol API for quotes on each token
5. Filter tokens by minimum output value (`minOut`)
6. Post orders to the CoW Protocol orderbook
7. Execute the `drip()` transaction with token approvals and swap details
8. Exit after completing one cycle
</Step>

## How It Works

The keeper bot performs the following operations:

### 1. Token Discovery

Based on the `--token-list-strategy`:

- **Explorer**: Queries block explorer APIs for current token holdings
- **Chain**: Scans Trade events from Settlement contract for last N blocks

### 2. Balance and Allowance Check

For each discovered token:

```typescript
// Fetch actual balances from Settlement contract
const balances = await getBalances(provider, settlement, tokenAddresses);

// Check current allowances to VaultRelayer
const allowances = await getAllowances(provider, settlement, vaultRelayer, tokenAddresses);

// Determine which tokens need approval
const needsApproval = allowance < balance;
```

### 3. Quote Filtering

Requests quotes from CoW Protocol orderbook API:

```typescript
const quote = await orderBookApi.getQuote({
  sellToken: token.address,
  sellAmountBeforeFee: balance.toString(),
  kind: OrderQuoteSideKindSell.SELL,
  buyToken: wrappedNativeToken,
  from: settlement,
});

// Apply slippage tolerance
const buyAmount = quote.buyAmount * (10000 - buyAmountSlippageBps) / 10000;

// Filter by minimum output
if (buyAmount >= minOut) {
  // Include in swap
}
```

### 4. Order Posting

Posts orders to CoW Protocol orderbook for all valid tokens.

### 5. Drip Execution

Calls the `drip()` function with:

- Tokens that need approval
- Swap details (token, sellAmount, buyAmount) for each order

<Info>
The keeper runs once and exits. For continuous operation, run it on a schedule (e.g., via cron or a task scheduler).
</Info>

## Troubleshooting

### Keeper Address Mismatch

**Error**: `The provided private key belongs to 0x..., which doesn't match the keeper (0x...)`

**Solution**: Ensure your `PRIVATE_KEY` environment variable corresponds to the keeper address configured in the module.

### Insufficient Funds

**Error**: Transaction fails due to insufficient gas

**Solution**: Ensure the keeper address has enough native tokens to pay for gas fees.

### No Tokens Found

**Issue**: Keeper reports 0 tokens after filtering

**Possible causes**:
- No tokens in Settlement contract meet the `minOut` threshold
- Token list strategy not finding tokens (try switching strategies)
- All tokens lack sufficient liquidity on CoW Protocol

**Solution**: Check token balances in the Settlement contract and verify `minOut` is not too high.

### RPC Rate Limiting

**Issue**: RPC calls failing or timing out when using `chain` strategy

**Solution**:
- Use `--rpc-url` to specify a dedicated RPC endpoint
- Reduce `--lookback-range` to scan fewer blocks
- Switch to `explorer` strategy if available

### Quote API Failures

**Issue**: Many tokens rejected during quote filtering

**Possible causes**:
- Tokens have insufficient liquidity on CoW Protocol
- Token is not tradeable (e.g., rebasing token, token with transfer fees)

**Solution**: This is normal behavior. The keeper filters out untradeable tokens automatically.
